教學指南 PWA

🛠️ PWA 工具集 Dashboard 建立教學

💡

核心目標： 將日常會用到嘅小工具集中喺同一個有設計感嘅地方，並支援 PWA 技術，令網站可以好似原生 App 咁安裝到手機主畫面。

1. 開發與設計思路

為咗配合「簡約、大地色系 (Earth Tones)」嘅要求，網站使用米白色 #f4f1ea 作為背景，配搭溫暖嘅棕色 #8b7355 作為主色調。為咗最快達成 MVP（最小可行產品），所有邏輯皆整合於單一 index.html 中，透過 DOM 切換模擬單頁應用 (SPA) 嘅流暢體驗。

2. 核心檔案結構

  • index.html： 主程式、UI 與所有工具邏輯。
  • manifest.json： PWA 應用程式清單，定義 App 名稱、顏色與圖示。
  • sw.js： Service Worker，提供離線快取支援。

3. 部署至 Vercel (免費上線)

  1. 登入 GitHub 建立 Public Repository。
  2. 將上述 3 個檔案拖曳上傳至最外層（Root directory）。
  3. 登入 Vercel，點擊「Add New Project」並從 GitHub 匯入。
  4. 點擊 Deploy，幾十秒後即可獲得真實網址。

關於MANUS（超詳細版）

💬

關於 Manus

官方首頁：https://manus.im/
官方文件首頁：https://manus.im/docs/introduction/welcome
官方價格頁：https://manus.im/pricing

Manus 係一個以 AI Agent 為核心定位嘅系統。
如果用最簡單一句去理解佢，同傳統 AI 最大分別就係：

傳統 AI 主要負責回答；Manus 主要負責執行。

官方文件將 Manus 描述為一個 autonomous general AI agent，重點唔係陪你對話，而係喺一個完整可操作嘅環境入面，幫你完成由規劃、執行到交付成果嘅整條工作鏈。

呢個定位其實反映咗 AI 產品一個幾關鍵嘅轉向：
AI 唔再只係一個「幫你諗」或者「幫你寫」嘅工具，而係開始變成一個可以代你處理工作流程、實際落手做事嘅代理人。

🤖

Manus 背後公司與核心人物

Manus 由 Butterfly Effect Pte. Ltd. 開發。現時公開資料可以比較穩陣確認到嘅核心人物包括：

  • 肖弘（Xiao Hong，別名 Red）｜CEO
  • 季逸超（Ji Yichao，常見別名 Peak）｜Chief Scientist
  • 張濤（Zhang Tao）｜聯合創辦團隊成員之一

其中肖弘喺官方內容同媒體報導中都係 Manus 最核心、最有代表性嘅人物；而「Red」呢個名亦唔係網民亂叫，係官方年度信入面直接用過嘅署名。季逸超則比較代表技術方向，因為 Manus 有篇好重要嘅技術文章〈Context Engineering for AI Agents〉，就係由佢署名發表。

公司背景方面，公開資料顯示 Manus 早期與中國團隊有關，而後期主要營運重心轉向新加坡。呢個轉移除咗係公司架構變化之外，亦反映咗 AI 公司喺地緣政治、資本市場同監管環境之下，需要考慮更複雜嘅國際部署。

💡

創辦理念：點解 Manus 唔似一般 AI 產品

如果只睇市面上好多 AI 產品，佢哋通常會圍繞幾個方向去講自己：更聰明、更快、更似人、更強大嘅模型。

但 Manus 有趣嘅地方係，佢對外講法冇咁集中喺「模型幾勁」，反而一直強調 action、doing、deliver results 呢啲字眼。

官方 About 頁有一句幾值得記住：

We don't want overachieving AI, we want overachieving Humans.

而官方年度信入面，肖弘亦講過一句更直接嘅：

AI shouldn’t just think, it should also do.

如果將呢兩句放埋一齊睇，其實可以幾清楚咁理解到 Manus 想做咩：
佢唔係想做一個「更識答你」嘅 AI，而係想做一個「真係幫你交到結果」嘅 AI。

💬

Manus 背後點樣運作？

▼ Manus 核心技術思路：唔係靠一個超級模型，而係靠 Agent 系統

Manus 嘅技術路線，並唔係宣稱自己訓練咗一個全新、超越所有人嘅端到端大模型。相反，官方技術文章公開講到，團隊早期有諗過兩條路：

  1. 訓練一個端到端 agent model
  2. 建立喺現有 frontier models 之上，再做 agent 系統

最後佢哋選擇咗後者，即係：
👉 重點唔係重新發明一個模型，而係做好「context engineering」同「execution system」。

呢條路線其實好重要，因為佢決定咗 Manus 唔係一個單純比拼 IQ 嘅產品，而係一個比拼：任務拆解能力、工具調度能力、上下文管理能力、執行穩定性、交付結果能力。
換句話講，Manus 代表嘅唔係「模型戰」，而係「系統戰」。


▼ Manus 係點運作：由 chat 變成一個會做嘢嘅循環系統

從技術上睇，Manus 可以簡化理解成以下循環：

  1. 理解需求： 由語言模型去理解你輸入嘅目標、限制、上下文。
  2. 規劃步驟： 將大任務拆成多個細步驟，例如要唔要先搵資料、要唔要先開某個工具、要唔要生成文件。
  3. 調用工具： 根據任務需要，去用唔同工具，包括 API、網站、connector、本地資源、shell command 等。
  4. 執行與觀察結果： 做完一步之後，系統會睇返結果，再決定下一步。
  5. 反覆迭代直到完成： 呢個就係典型 agent loop：唔係一次答完，而係一路做一路修正。

官方技術文章好清楚講到，呢個 loop 其實係由「action → observation → next action」一路推進，而唔係普通 chat model 嗰種一來一回。


▼ 沙盒虛擬電腦：點解 Manus 唔止係聊天介面

另一個非常關鍵嘅理解位係：Manus 嘅真正工作場所唔係 chat box，而係一個 sandbox 虛擬電腦環境。

官方文件形容得好明確：佢唔只係一個對話 UI，而係運行喺一個完整虛擬電腦入面，當中包括：網絡連線、持久檔案系統、可安裝軟件、可建立工具、可執行程式碼。

所以佢做到嘅，就唔再係「答你應該點做」，而係真係可以：

  • 幫你建立報告
  • 幫你整理資料
  • 幫你產出網站
  • 幫你跑指令
  • 幫你生成交付物

呢個「虛擬電腦」概念，係 Manus 能夠由 AI 助手變成 AI Agent 嘅根基。


▼ 點解 Manus 可以做到低門檻

如果只係講一句「因為佢自然語言操作」其實太淺。真正原因冇咁簡單。Manus 之所以低門檻，唔係因為用戶突然變得識得寫 prompt，而係因為佢將原本最難、最工程化嗰部分——即環境、工具、權限、流程——幫你包埋。

低門檻其實來自三樣嘢：

  1. 將執行環境抽象化： 你唔需要自己 set 好本地 IDE、CLI、依賴環境，因為系統已經有一個可執行環境。
  2. 將工具使用標準化： 透過 MCP Connectors、Browser Operator 等，外部工具唔再需要逐個自砌 integration。
  3. 將流程變成系統能力： Skill、Scheduled Tasks 呢啲唔係單一功能，而係將「一次性操作」變成「可重用流程」。

所以 Manus 真正降低嘅唔係「用字門檻」，而係 系統使用門檻。

💬

Manus 四個最重要功能模組

如果要理解 Manus 點樣由「會講」走到「會做」，其實最值得睇係以下四個模組。

▼ MCP Connector
MCP Connector 係 Manus 打通外部世界嘅方法。官方文件寫到好清楚，連接之後 Manus 可以：讀取資料、執行動作、協調跨平台 workflow。例如連接 Gmail、Notion、Telegram 之後，AI 就唔再停留喺「模擬」，而係可以真係對住你帳戶資料做嘢。

▼ Skill
Skill 係另一個好重要嘅概念。佢本質上係將一套經驗、SOP、工具用法打包成可重用模組。官方描述包括：SKILL.md, scripts, best practices, reusable resources。你可以將一個工作流固定落嚟，之後唔使每次重新解釋。對內容創作者嚟講，呢樣非常值錢，因為研究、整理、格式化輸出本身就係一套可被標準化嘅流程。

▼ Scheduled Tasks（Cron）
Manus 官方唔直接叫佢 Cron Job，而係叫 Scheduled Tasks。本質上一樣，即：每日自動做、每週自動做、每月自動做（例如 daily digest、weekly tracking、monthly report）。呢個功能令 AI 唔再係「你要用先打開」，而係變成可持續運作嘅系統。

▼ My Computer
My Computer 係近一步拉高 Manus 層次嘅功能。佢唔只係 cloud agent，而係開始可以接觸本地電腦：讀寫檔案、用 CLI、操作本地 app、幫你整理 folder。從產品角度睇，呢個功能係一個重要轉折點，因為代表 Manus 正式由「雲端任務型 agent」走向「本地工作型 agent」。

▼ Free Plan

  • $0 / 月
  • 每日約 300 credits
  • 1 個 concurrent task
  • 2 個 scheduled tasks

▼ Pro Plan

  • $20 / 月起（另有 $40/月 級別，並提到有 7-day free trial）
  • 每月約 4,000–8,000 credits 起跳

▼ Team Plan

  • $20 / seat / 月起
  • 包括 team analytics、SSO、data training opt-out 等

但最重要唔係 plan 名，而係 「Manus 係 credit-based system」。
即係唔同任務、唔同複雜度，會消耗唔同 credits。佢唔係傳統「付咗月費就無限任用」嗰種產品。

仲有 WebDev / Deployment 額外成本

如果你用 Manus 做嘅唔止係研究、報告，而係：
▶️ 部署網站、用 database、用 cloud hosting、用外部 API
咁就會進入 usage-based 成本範圍。

官方 Help Center 都明確區分：subscription credits 與 WebDev usage balance。

所以最準確理解係：
Manus = 訂閱費 + credits 消耗 + 雲端部署 / API 用量

對內容創作者嚟講，如果你主要係做 research、report、slides、一般 workflow，主要成本係 credits。但如果你開始用佢整 web app、長期運作系統，就唔可以淨係睇月費。

案例Level 1｜📩 自動處理 Email

💡

案例功能做到什麼？

👉 自動幫你讀取 Email，並生成合適回覆（例如禮貌拒絕合作）

🔥

案例重點

  • 不用逐句思考語氣，AI會幫你整理好
  • 可以處理「想拒絕但又要禮貌」的情境
  • 支援英文 Email，自動翻譯＋回覆
  • 可進一步用於分類、整理 Email
🧠

案例涉及MANUS的功能

  • 連接外部帳戶（MCP Connector）
  • 可讀取實際 Email 內容（Connected Data Access）
  • 可理解郵件語境並生成回覆（AI Content Generation）

🖼️ 操作流程

1. 點擊「連接應用程式」（MCP Connector）

2. 連接 Gmail 帳戶

3. 確認 AI 能讀取你的 Email

4. 輸入需求（例如：幫我回覆呢封合作 email）

5. Manus 自動生成回覆並存入草稿

💬 我實際用嘅 Prompt（隨意口語版）

我想你幫我DRAFT一封中文回覆郵件放草稿箱，係話多謝合作詢問，但可惜時間不適合（其實係冇乜興趣），用輕鬆又帶點FORMAL的格式回應，用DEAR XX開頭，結尾用BEST REGARDS

💡 使用小貼士

  • 唔需要寫好正式 prompt，用口語都可以
  • 可以叫佢再改語氣（再禮貌啲 / 再直接啲）
  • 可以延伸用法：
    • 幫你整理 Inbox 重點
    • 幫你分類郵件
    • 幫你翻譯＋總結英文 Email

💡

案例功能做到什麼？

👉 每日自動幫你整理 AI 新聞，並定時發送到 Telegram（完全自動化）
由「用 AI」→ 升級為「讓 AI 每日幫你工作」

🔥

案例重點

  • 由「一次性使用」變成「每日自動運作」
  • 可同時追蹤中英文 AI 新聞來源
  • 自動篩選熱門內容（Top 5）
  • 可加入總結＋翻譯（香港口語）
  • 可設定固定輸出格式（穩定產出）
🧠

案例涉及MANUS的功能

  • 可設定定時自動任務（Cron Job）
  • 可建立固定執行流程（Skill）
  • 可整合多來源資訊（Multi-source Aggregation）
  • 可自動整理＋輸出內容（AI Content Generation）
  • 可發送至外部平台（Integration / Automation）

🖼️ 操作流程

1. 建立每日日報需求（先簡單講清楚內容）

2. 設定 Telegram Bot（接收訊息）

3. 與 Manus 逐步調整輸出內容

4. 將整個流程封裝為 Skill

5. 每日自動觸發並發送內容 (左: 每日香港新聞 / 右: 每日AI日報)

🌟 Prompt（每日AI日報の詳細完整版）

幫我建立一個每日香港時間23:55傳給我的 AI 新聞速報系統，腳本要永久儲存喺 Manus Skill 目錄，確保 CRON 每次都能穩定執行。 **新聞來源：** - 英文：The Verge + TechCrunch - 中文：36氪 + 量子位 **篩選邏輯：** 抓取香港時間當日 00:00 後嘅所有文章，用 GPT 語意聚類，按話題被多個來源提及嘅次數排序 選出最 HIT 嘅英文 Top 5 及中文 Top 5（唔用 keyword 過濾，直接交畀 GPT 判斷）。 **訊息格式：** 「 ━━━━━━━━━━ AI 日報 | [日期] ━━━━━━━━━━ 【 今日重點】 [分段總結，約150字，香港廣東話書面語，段與段之間空一行] ◠◡◠◡◠◡◠◡◠◡◠◡ 【 英文 Top 5】[英文媒體名稱] [編號] [英文標題] [繁體中文翻譯] 時間：[ET] | [HKT] [連結] [約100字香港口語解釋，專業術語括號保留英文] ◠◡◠◡◠◡◠◡◠◡◠◡ 【 中文 Top 5】[中文媒體名稱] [編號] [繁體中文標題] [ X 個來源提及（如有）] 時間：[HH:MM] [連結] [約100字香港口語解釋，專業術語括號保留英文] ◠◡◠◡◠◡◠◡◠◡◠◡ 每日自動更新 」 **語言要求：** - 全部用香港廣東話書面語 - 中文標題自動轉繁體 - 專業術語括號保留英文原文，例如「大型語言模型（LLM）」 **Telegram 設定請適當時候再詢問我** 完成後： 1. 將完整腳本儲存到 Skill 目錄：/home/ubuntu/skills/ai-news-bot/scripts/ai_news.py 2. 建立 SKILL.md 說明執行步驟 3. 先手動測試一次，確認 Telegram 收到正確格式 4. 確認後設定每日 CRON 排程

💡 使用小貼士

  • 唔需要一開始寫完整 prompt，可以一路同 AI refine
  • 記住「1 個 Cron = 1 個 Skill」，會穩定好多
  • 如果唔用 Skill： 每次執行可能格式會變／漏步驟
